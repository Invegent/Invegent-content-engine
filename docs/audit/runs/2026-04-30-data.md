## F-2026-04-30-D-001  ·  MEDIUM  ·  closed-action-taken

**Role:** Data Auditor  
**Raised:** 2026-04-30 UTC (audit run: runs/2026-04-30-data.md)  
**Area:** Index coverage  
**Object:** `m.ai_job`

### Issue

`m.ai_job` has two UNIQUE indexes over the same column tuple: `(post_draft_id, job_type)`. Because `m.ai_job` is a hot operational table used by the slot/AI job path, duplicate uniqueness enforcement risks unnecessary write overhead and increases migration/constraint ambiguity.

### Evidence

- Snapshot section 15 lists both `ux_ai_job_post_draft_job_type UNIQUE (post_draft_id, job_type)` and `ux_ai_job_unique UNIQUE (post_draft_id, job_type)` under `m.ai_job`.
- Snapshot section 8 shows `m.ai_job` is active in the last 7 days, with both legacy `rewrite_v1` and shadow `slot_fill_synthesis_v1` jobs present.
- The Data Auditor role explicitly treats `m.ai_job` as a hot table and includes index coverage on `(post_draft_id)` / queue paths in scope.

### Recommended action

Confirm whether both indexes are backing separate constraints or whether one is redundant. If redundant, keep the canonical constraint/index name and drop the duplicate via a migration. If both are intentionally retained, document why in the relevant run state or decision file.

### Resolution

**closed-action-taken 2026-04-30 evening (chat closure session).**

Verification before action:

- Both indexes confirmed as bare UNIQUE indexes (`pg_class.indisunique=true`) over identical column tuple `(post_draft_id, job_type)` with no partial WHERE clauses.
- `pg_constraint` query confirmed **neither index backs a named UNIQUE constraint** — both are redundant pure unique indexes.
- The `ON CONFLICT (post_draft_id, job_type)` clause in `m.fill_pending_slots` resolves against the column tuple, not against an index name — so dropping either is safe; surviving index continues to satisfy the UPSERT.
- The 27 Apr migration `stage_12_053_fill_pending_slots_ai_job_upsert` (commit prior to today) explicitly noted the redundancy and deferred cleanup to a separate migration.

Action:

- Migration `audit_drop_redundant_ai_job_unique_index` applied 2026-04-30 evening via Supabase MCP per D170.
- Dropped: `m.ux_ai_job_post_draft_job_type` (the more verbose duplicate added later).
- Kept: `m.ux_ai_job_unique` as canonical (named in the original 27 Apr migration prose; shorter / more general).
- Pre/post DO block verification: index count on `m.ai_job` 6 → 5 (-1 as expected). Canonical index retained.
- UPSERT path preserved end-to-end.

This closes the deferred cleanup explicitly named in the 27 Apr migration's comment block. Severity-honest assessment: this finding could have been HIGH given the explicit deferral note in the prior migration; auditor's MEDIUM rating was conservative.

## F-2026-04-30-D-002  ·  MEDIUM  ·  closed-action-pending

**Role:** Data Auditor  
**Raised:** 2026-04-30 UTC (audit run: runs/2026-04-30-data.md)  
**Area:** Index coverage  
**Object:** `m.slot`

### Issue

`m.slot` is missing an index on `filled_draft_id` in the snapshot's hot-table index inventory. The Data Auditor role names `m.slot` as a hot table and expects index coverage on `(client_id, platform, scheduled_publish_at)`, `(status)`, and `(filled_draft_id)`. The first two patterns appear covered; `filled_draft_id` does not.

### Evidence

- Snapshot section 15 lists these `m.slot` indexes: `idx_slot_client_platform_scheduled`, `idx_slot_fill_in_progress`, `idx_slot_pending_fill`, `idx_slot_status_window`, `idx_slot_unique_active`, and `slot_pkey`.
- None of the listed indexes are on `filled_draft_id`.
- Slot state is active in section 7 across all four clients, and failed/filled/future slot states are present, so reverse lookups from generated/filled draft to slot are plausible operational paths.

### Recommended action

Check production query paths for lookups from `m.post_draft` / filled draft back to `m.slot`. If that lookup exists or will be used in Phase C, add an index on `m.slot(filled_draft_id)` via migration. If the column is not used for lookup, document that decision and consider removing this expectation from the Data Auditor role.

### Resolution

**closed-action-pending 2026-04-30 evening (chat closure session).**

Verification before deciding:

- Auditor's structural concern is correct: there is no index on `m.slot.filled_draft_id`. Confirmed via direct `pg_index` query.
- Production code DOES use `filled_draft_id` in lookup patterns: `m.fill_pending_slots`, `m.handle_draft_rejection`, `m.recover_stuck_slots` — all three use `WHERE filled_draft_id = ...` or join patterns.
- HOWEVER: `m.slot` currently has only **159 live tuples**. At this size, Postgres planner correctly chooses sequential scans over index scans — adding an index now would not improve performance and might be ignored.
- `pg_stat_user_tables` shows m.slot at 97.1% idx_scan_pct already (8814 idx vs 263 seq), meaning existing indexes ARE being used for the queries that benefit from them. The seq scans are likely the `filled_draft_id` lookups, but they're trivially fast at this row count.
- Phase C cutover will increase row volume substantially as more clients onboard. The threshold where btree on a uuid column beats seq scan is roughly 5,000-10,000 rows.

Action:

- New backlog item B19 added to `docs/00_action_list.md`: add `idx_slot_filled_draft_id` migration when `m.slot` row count exceeds 5,000 OR when EXPLAIN on production-relevant query plans starts showing seq scan as the chosen path with measurable cost (whichever fires first).
- Trigger query for promoting B19 to Ready: `SELECT n_live_tup FROM pg_stat_user_tables WHERE schemaname='m' AND relname='slot'`.
- Auditor's role expectation for `m.slot.filled_draft_id` index is retained — the audit catch is correct in principle even though the production impact is currently zero.

Forward note: a healthier signal than "role expects index X exists" would be "role expects index X exists OR row count is below indexing threshold OR EXPLAIN plan is satisfactory". Future audit role refinement could add row-count-aware thresholds to the index expectations. Captured for next role refinement.
