Use either as a one-shot manual run (during the day, observed) or as a scheduled task via Cowork's `/schedule` UI (overnight, after first run is verified).\n\n**Spec reference:** `docs/runtime/automation_v1_spec.md` (D182 v1 locked)\n\n---\n\n## How to use this\n\n### Manual one-shot (recommended for first run)\n\n1. Laptop must be awake AND Claude Desktop open at the scheduled time \u2014 same constraint as `openclaw tui`\n\n---\n\n## The prompt (paste this into Cowork)\n\n```\nYou are executing a brief from the ICE non-blocking execution system (D182 v1).\n\nYour job for this run:\n\n1. Read docs/briefs/queue.md from github.com/Invegent/Invegent-content-engine via GitHub MCP.\n2. Find the FIRST row in the Active queue table with status: ready.\n3. If none, write a state file noting \"no ready briefs\" and stop.\n4. Read the matching brief at docs/briefs/{brief_id}.md \u2014 this is your full instruction set.\n5. Verify the brief's frontmatter is complete: brief_id, status, risk_tier, owner, default_action, allowed_paths, forbidden_actions, idempotency_check, success_output. If any required field is missing, halt and report.\n6. Run the brief's idempotency_check. If it indicates the work is already done, write a state file noting \"already_applied\" and stop.\n7. Execute the brief end-to-end per its instructions. Follow these rules without exception:\n\n   - You may write files only inside the paths listed in allowed_paths.\n   - You must NOT take any action listed in forbidden_actions. Specifically: never call apply_migration, never UPDATE/INSERT/DELETE production data, never delete branches, never merge PRs, never close audit findings.\n   - When you hit a decision point not pre-answered in the brief's \"Likely questions and defaults\" section, write the question to docs/runtime/claude_questions.md using the format in that file, document your default in the state file, and proceed with the default. Do NOT block waiting for an answer.\n   - When the brief's \"Likely questions and defaults\" pre-answers a decision, use the default unless evidence in the run contradicts the brief.\n   - For the first test run (Phase D ARRAY mop-up), the brief's pre-flight findings were captured by PK before authoring \u2014 do not re-query Supabase for data the brief already provides. If a value seems wrong, write a question and use the brief's data anyway.\n   - If you encounter a Tier 2 (production-affecting) or Tier 3 (judgment-heavy: brand voice, client-facing wording, legal/compliance, secrets) question, halt that scope, write ESCALATION_REQUIRED in the state file under \"Stop conditions\", and stop.\n\n8. Write the per-run state file at docs/runtime/runs/{brief_id}-{ISO-timestamp}.md following docs/runtime/state_file_template.md. Specifically:\n\n   - Status: end at review_required if work succeeded; failed if it didn't; blocked if Tier 2/3 escalation hit.\n   - Work completed: explicit list of files created or updated, with paths.\n   - Questions asked: every question ID written to claude_questions.md.\n   - Validation results: N/A on first run (Phase 4 of D182 not yet built).\n   - Stop conditions: ESCALATION_REQUIRED if Tier 2/3, otherwise none.\n   - Needs PK approval: explicit \"PK does X to advance this brief\".\n   - Token usage: report your started/ended token figures.\n   - Issues encountered: anything unexpected even if recovered.\n   - Next step: who handles what next.\n\n9. Commit the brief output (migration file, state file, any answer-key follow-ups) directly to main with a clear commit message. Do not open a PR; D165 standing rule is direct-push to main for non-risky single-repo work.\n\n10. Write \"running\" then later \"review_required\" in the brief's frontmatter status as you transition. Use the GitHub MCP get_file_contents + create_or_update_file pattern with fresh SHA.\n\n11. Update docs/briefs/queue.md to reflect the new status of the brief and the latest run timestamp.\n\n12. Stop. Do not start a second brief in the same run \u2014 even if another shows status: ready, stop after the first to keep observation signal clean.\n\nIf at any point you are uncertain about whether an action is allowed:\n- If it touches production data, the answer is no. Stop and escalate.\n- If it's a configuration or text edit inside an allowed path, the answer is yes.\n- If it's between, write the question to claude_questions.md and proceed with the default-and-continue rule.\n\nReport back when done with: brief_id processed, status, files created, token usage. PK reviews in the morning.\n```\n\n---\n\n## Notes on first run\n\n**Don't schedule it overnight first.** Run it manually during a window when you're at the laptop. Watch the run unfold. No silent retries.\n\n## Sunset review\n\nIf this prompt is still in use unchanged on 12 May 2026, that's a smell.
