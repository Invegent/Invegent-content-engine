### Migration file design — idempotency strategy

**File 1: `supabase/migrations/20260508003500_f_heygen_rpc_migrations_missing.sql`** (5487 B, 123 lines)
- All 5 functions use `CREATE OR REPLACE FUNCTION` (idempotent — replays no-op against production)
- Functions reference `c.brand_avatar` + `c.brand_stakeholder`. plpgsql does
  NOT validate table refs at CREATE — succeeds even on fresh DB before c-schema
  tables exist (assumed pre-req for fresh-DB rebuild order).
- Grants: PUBLIC EXECUTE is Postgres default — no explicit GRANT statements.

**File 2: `supabase/migrations/20260508003600_f_insights_rpc_migrations_missing.sql`** (7307 B, 163 lines)
- `CREATE TABLE IF NOT EXISTS` for `m.post_format_performance` with inline PK + UNIQUE
- `CREATE INDEX IF NOT EXISTS` for `idx_format_perf_client`
- `ALTER TABLE … ADD COLUMN IF NOT EXISTS` for `m.post_draft.recommended_format`
- 2 FK adds wrapped in `DO $$` blocks with target-existence + constraint-absence
  guards (Postgres has no `IF NOT EXISTS` for `ADD CONSTRAINT`)
- `CREATE OR REPLACE FUNCTION` for `upsert_format_performance` (last in file)

### Safety verification
- Self-grep for DROP / DELETE / TRUNCATE / REVOKE at file scope: zero matches
- UPDATE statements appear ONLY inside function body definition text

- Both files traversed self-check for idempotency markers and destructive SQL
