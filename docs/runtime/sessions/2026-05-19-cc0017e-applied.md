## 6. Five brief defects surfaced (Path B-prime correctives)

Each surfaced as a hard-stop event to PK; PK directed Path B-prime in each case.

### 6.1 Defect 1 — severity='low' invalid
- **Where:** vchecks.md V-D-setup SQL example
- **Problem:** `case_severity_check` accepts only `{info, warn, critical}`; `low` violates
- **Path B-prime:** runtime substitute `severity='info'` in fixture seed
- **v1.1 doc patch:** vchecks.md V-D-setup section — severity value correction

### 6.2 Defect 2 — category='cc-0017e/v-d/category' violates FK
- **Where:** vchecks.md V-D-setup SQL example
- **Problem:** `case_category_fkey` requires presence in `friction.category` (6 valid codes)
- **Path B-prime:** runtime substitute `category='unclassified'` in fixture seed
- **v1.1 doc patch:** vchecks.md V-D-setup section — category value correction
- **Shared root cause with defect 1:** V-D fixture authoring did not probe full constraint surface (FK + CHECK) on `friction."case"`. New L-v2.90 candidate.

### 6.3 Defect 3 — fn_triage_case CREATE OR REPLACE arity-change semantics
- **Where:** migration-sql.md §2 + risks-and-grants.md R4
- **Problem:** Brief stated `p_actor DEFAULT NULL` makes the patch "signature-compatible". True for **call resolution** but missed PostgreSQL's overload-on-arglist semantics: `CREATE OR REPLACE FUNCTION` with a different arity creates a sibling overload, not a replacement. After main migration, BOTH the legacy 10-arg AND the new 11-arg fn_triage_case existed. Legacy 10-arg retained its pre-patch body (no case_history INSERT, no triaged_at/triaged_by write logic). Defeats item C compat patch intent for any external caller passing 10 args positional.
- **Path B-prime:** explicit `DROP FUNCTION IF EXISTS friction.fn_triage_case(uuid, text, text, boolean, text, text, text, timestamptz, text, text)` corrective migration. Post-DROP: only 11-arg version remains.
- **v1.1 doc patch:** migration-sql.md §2 (add explicit DROP statement before CREATE OR REPLACE) + risks-and-grants.md R4 (correct rationale)
- **NEW L-v2.90-b candidate:** CREATE OR REPLACE FUNCTION with arity change requires explicit prior DROP of the previous signature.

### 6.4 Defect 4 — V-D fixture naming mismatch with purge_test_case regex
- **Where:** vchecks.md V-Z1 cleanup spec + V-D-setup fixture titles
- **Problem:** `purge_test_case` enforces pattern format `^cc-[0-9]{4}[a-z]?-test/`. Brief used `cc-0017e/v-d/...` titles, which would error with P0001 `must match...`. Slash-prefix convention correct (cc-0017d v1.1 Drift 3 reaffirmation) but suffix `-test/` segment after brief code missed.
- **v1.1 doc patch:** vchecks.md V-D-setup → rename to `cc-0017e-test/fixture-001..004` form; V-Z1 cleanup spec to match.
- **NEW L-v2.90-c candidate:** V-D fixture naming must conform to purge_test_case helper regex; slash-prefix alone is insufficient.

### 6.5 Defect 5 — purge_test_case helper case_history coverage gap
- **Where:** Helper realised v2.86; case_history introduced v2.90 (this session). Helper not updated.
- **Problem:** `purge_test_case` deletes from `event`, `case`, `emit_error` — NOT case_history. With case_history.case_id FK ON DELETE RESTRICT, calling the helper on any case with case_history rows would FK-fail at the case DELETE step.
- **Path B-prime:** inline cleanup migration `cc_0017e_vcheck_audit_cleanup` with explicit dependency-ordered DELETEs (case_history → event → emit_error → case).
- **v1.1 doc patch + future brief candidate:** helper extension to handle case_history is a future Wave candidate; vchecks.md cleanup section needs cleanup-pattern alignment with the actual deletable scope.
- **NEW L-v2.90-d candidate:** when introducing a shadow table with FK ON DELETE RESTRICT, audit all existing cleanup helpers for coverage gaps at brief authoring time.

### 6.6 Defects 6 + 7 — close-the-loop template phantom columns
- **Where:** d01-postapply-deferred.md §4 close-the-loop UPDATE templates
- **Problem:** Template references `resolved_at` AND `result_summary`. **Neither column exists** on m.chatgpt_review. Actual schema has `id, status, action_taken, resolved_by, escalated_at, escalation_resolved_at, error_message, escalation_reason` (no `resolved_at`; no `result_summary`).
- **Disposition:** non-blocking — closed using schema-correct columns (status='resolved', action_taken='applied_with_correction', resolved_by='cc-0017e-close-v2.90'). Rich result summary belongs in this session file anyway, not in m.chatgpt_review.
- **v1.1 doc patch:** d01-postapply-deferred.md §4 — all 4 template variants need column-name correction.
- **NEW L-v2.90-e candidate:** close-the-loop SQL templates must be validated against actual m.chatgpt_review schema, not assumed columns.

### 6.7 Defect 8 — risks-and-grants.md §3 broad-scope overclaim
- **Where:** risks-and-grants.md §3 lockdown verification clause
- **Problem:** Brief states "authenticated + anon have zero grants on friction.*" but 6 SELECT grants exist on reference/config tables (`category`, `emission_rule`, `emission_rule_history`, `experiment_run`, `notification_policy`, `source`).
