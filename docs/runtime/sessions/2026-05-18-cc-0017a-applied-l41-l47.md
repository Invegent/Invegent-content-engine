### 6.3 L-v2.81-a — NEW CANDIDATE PROPOSED (1 occurrence)

**Pattern: Parallel-session apply coordination.**

When a Path B clearance is granted (apply deferred to a separate session), it can be acted on by ANY parallel Claude session with Supabase MCP access — across projects, across Claude surfaces (web / desktop / mobile / API client), across browser tabs, across accounts that share Supabase access. The chat-side audit trail (`sync_state` in any one project) cannot reliably observe parallel-session applies. Therefore:

1. **Operator discipline:** After granting a Path B clearance, the operator should commit to applying from a single designated chat (preferably the one where D-01 was cleared) and avoid spawning parallel sessions with apply rights. If a parallel session DOES apply, the operator should immediately notify the originating session so it can update its sync_state.
2. **Next-session discipline:** Any session reading sync_state with an OPEN apply gate MUST run empirical pre-flight before assuming the gate is still open. A parallel session may have already applied. **The full pre-flight P-set is the authoritative source of truth; sync_state is best-effort.** This subsumes the L41 baseline.
3. **Applying-session discipline:** Post-apply, the applying session must commit an audit-trail update to the originating project's sync_state — OR notify the operator to do so via the originating chat. If neither happens, the next session inherits a provenance gap that must be investigated via postgres logs (`application_name=mgmt-api` connection signature) + migration history (`supabase_migrations.schema_migrations`) + `m.chatgpt_review` (D-01 clearance trail) + `conversation_search` + `recent_chats`.

**Promotion-eligible on re-exercise.** Baseline pattern would govern parallel-session discipline + next-session pre-flight discipline.
