## Workstream 1 ‚Äî Slotting cost-waste audit + cc-0019 brief

**Audit (read-only) findings:**
- Token-spend trigger is `m.fill_pending_slots` (jobid 75, `*/10`): on a fill it INSERTs a skeleton `m.post_draft` (`approval_status='draft'`) + a `queued` `m.ai_job` (`slot_fill_synthesis_v1`, priority 100). The **ai-worker** (jobids 5/7) then calls the model = tokens burned.
- `fill_pending_slots` gates only on dedup/verticals/signal-pool fitness/format policy/pool health/evergreen ‚Äî **no `publish_enabled`/platform/paused/cron check anywhere.**
- All cost-protecting gates (`publish_enabled`, active, `paused_until`, throttle, cron) live at the **publish layer**, 3 layers downstream of token spend.
- Measured waste (7d): 10 ai_jobs on `publish_enabled=false` pairs (NY IG 5 + PP IG 5, all succeeded) ‚âà 12% of ~81 ai_jobs; recurring every fill cycle. Fresh `filled` IG slots dated 2026-05-25+ confirmed for NY/PP post-void.
- Bonus finding (read-only, `pg_get_functiondef('m.materialise_slots')`): `materialise_slots` **does** reference `publish_enabled` ‚Äî but only inside the per-platform *format-preference* lookup. A disabled profile yields NULL preferred format ‚Üí empty `format_preference` ‚Üí the slot is **still inserted** (later defaults to `image_quote`). So disablement was wired to *format selection*, not *skip*. The eligibility flag existed but had the wrong effect ‚Äî explains why this was missed: misplaced gate, not a blind spot. The two switches were never unified: publishing = `client_publish_profile.publish_enabled`; materialisation = `client_publish_schedule.enabled` (NY/PP IG have `publish_enabled=false` but schedule `enabled=TRUE`). Shadow-mode cutover also normalised "generate-but-don't-publish," and the design modelled `shadow‚Üílive` but never `live‚Üídisabled` (which is what the 2207051 IG containment did weeks later).

**cc-0019 brief:** `docs/briefs/cc-0019-publish-eligibility-gate.md`
- Created at commit `81cb414` (13,582 bytes).
- Hardened at commit `01a6cdf` (16,057 bytes) ‚Äî added Design-detail ¬ß4 "Eligibility transition handling" (4 cases: eligible‚Üíineligible after fill / race before insert / job already completed / ineligible‚Üíeligible).
- Design: `m.is_publish_eligible(p_client_id, p_platform)` SECURITY DEFINER boolean (publish_enabled AND active AND not-paused AND client active; false on missing profile) + skip-gate at top of `fill_pending_slots` FOR-loop (writes `slot_fill_attempt` decision='skipped' skip_reason='publish_path_disabled', sets slot status, CONTINUE ‚Äî no draft, no ai_job) + ai-worker preflight (abandon `queued` job before model call if ineligible). Cron dimension deliberately out of v1 (handled operationally: set `publish_enabled=false` to also stop generation).

## Workstream 2 ‚Äî YouTube full-depth audit (root cause)

**Ground truth (m.post_publish, platform=youtube):** PP published 11 total, last **12 May 07:45 UTC**; NY published 2 total, last **11 May 09:45 UTC**. Invegent + CFW: zero (never connected).

**Re-break timeline:** failures resumed the day after each last success ‚Äî PP first failure 13 May 07:00, NY 12 May 09:00 ‚Äî all `invalid_grant: Token has been expired or revoked`. Windowing to post-5 May reconnect: **NY token lasted 7.00 days, PP 7.92 days.**

**Token state (c.client_channel.config):** NY + PP both hold a genuine refresh token (prefix `1//`, 103 chars), `token_updated_at` 5 May. **No expiry field stored at all** ‚Äî config keys are only `channel_id, channel_title, refresh_token, token_updated_at`.

**OAuth flow (invegent-dashboard):**
- `app/api/youtube/auth/route.ts` ‚Äî correctly sets `access_type=offline` + `prompt=consent` (forces refresh token on every reconnect). Scopes: `youtube.upload` + `youtube.readonly` (sensitive scopes).
- `app/api/youtube/callback/route.ts` ‚Äî exchanges code, stores `refresh_token`, and **hardcodes** `p_token_expires_at = now()+5 years` with the comment *"YouTube refresh tokens don't expire ‚Äî set 5 years out so dashboard shows Valid."* ‚Üí this is the source of the dashboard's fake "Expires 5 May 2031" / "24 May 2031" display. Cosmetic; the publisher never reads it.

**Root cause (definitive):** the OAuth app was in **Testing** publishing status. Google caps Testing-mode refresh tokens at exactly 7 days. Two independent accounts dying at 7.00/7.92 days = the fingerprint. The callback's "tokens don't expire" assumption is true for a *Production* app, false for a *Testing* one ‚Äî which is why the code, the dashboard, and every prior session assumed Production while the app was never published. Reconnecting only restarts the 7-day clock (the loop PK had been in: April break ‚Üí 5 May reconnect ‚Üí 12/13 May break).

**PK's hypothesis (recorded):** "token isn't the issue; the expiry wasn't recorded correctly." Half-right ‚Äî the recorded expiry IS fake (hardcoded 5y) ‚Äî but it's cosmetic and not the failure cause; the stored token is a valid `1//` refresh token and the publisher uses it directly. Cause is Google-side revocation, not storage.

## New findings / lesson candidates

- **F-YT-OAUTH-TESTING-MODE (root cause, RESOLVED):** YT broke because the OAuth app was in Testing publishing status ‚Üí 7-day refresh-token expiry. Fix = publish app to Production. Verified by two live publishes. **The cause had never been recorded before ‚Äî only the symptom (reconnect the token), which is why it recurred. Now recorded so it is not re-diagnosed as a token reconnect.** Durability (token surviving past ~31 May) to be spot-checked.
- **F-YT-FAILED-NO-RETRY (reliability gap, OPEN ‚Äî candidate brief):** youtube-publisher marks a *transient* auth failure as `video_status='failed'`, a *terminal, non-retryable* state invisible to its own SELECT. A token outage therefore silently bricks every rendered video until a manual backlog rescue. Fix candidate: distinguish retryable (`upload_failed_retryable`) vs terminal failure, or a retry sweep that re-promotes `failed`-with-video drafts. Would make YT self-heal after a future token blip.
- **F-YT-EXPIRY-DISPLAY-FAKE (cosmetic, OPEN ‚Äî candidate cleanup):** callback hardcodes `token_expires_at = now()+5y` ("tokens don't expire"); dashboard shows a fabricated "Valid until 2031/2031" that actively masks dead tokens. Worth recording reality (or omitting the field) so the display stops misleading.
- **F-YT-CALLBACK-NULL-TOKEN (latent):** `p_refresh_token: refresh_token ?? null` could wipe a good token if a reconnect ever returns null; `prompt=consent` should prevent it, but it's a sharp edge.
- **Slotting cost gap (cc-0019, brief ready):** publish-eligibility not checked before token spend ‚Äî see Workstream 1.
