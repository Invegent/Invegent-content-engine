## Two findings discovered during F-AAP-002 verification

### F-AAP-007 (NEW, P2) — audit Check 8 doesn't account for F-PUB-010 backpressure

Full audit run during F-AAP-002 verification surfaced Check 8 (`approved_drafts_missing_queue_entry`) returning **fail cnt=56** with detail "Approved drafts > 24h without queue entry: 56".

**Pattern:** Check 8's gate logic is "approved + no queue entry + >24h old = broken". But under F-PUB-010 hard-cap backpressure (introduced in publisher v1.6.0 et al.), drafts intentionally hold in `approved` status when (client, platform) queue is at cap. Per memory, the 5 over-cap combos drain via publish rate by design — not a system break.

**Severity:** P2. Same family as F-AAP-002 but for a different architectural change (F-PUB-010 backpressure, not slot-driven v4). Currently misreporting the system as broken; will grow as F-AAP-001's drain continues (the 23 v4 approvals at 09:30 UTC mostly landed on at-cap combos and are sitting in `approved`-without-queue waiting for queue space).

**Fix candidate:** `AND NOT EXISTS (cap-blocked check)` added to the gate, mirroring F-AAP-002 fix shape but for F-PUB-010. Concrete logic: exclude approved drafts where the (client, platform) publish queue is at or over `max_queued_per_platform` cap.

### B-AUDIT-CHECK5-DRIFT (NEW, P3) — confirmed silent drift

CC pre-flight #4 noted Check 5 (`stuck_publish_queue_items`) queries `m.post_publish_queue` for `status='locked'` which isn't in the current vocabulary (queued/published/dead/throttled/held/skipped). Lock state is signalled via `locked_at`/`locked_by` columns, not the status field.

**Pattern:** Older drift, not v4-specific. Check returns `pass` (cnt=0) regardless of actual stuck items — silent false-negative.

**Fix candidate:** rewrite Check 5 gate to query `locked_at IS NOT NULL AND locked_at < now() - interval '2 hours'` instead of `status = 'locked'`.
