## Q-005 root cause + Option A

**Root cause (recorded at v3.1):** brief v3.0 §12.2 built each finding object **without a `condition_key` field**. The live `friction.fn_emit_health_check_findings` therefore fell back to its `problem_key → condition_key` auto-deriver, which recognises ONLY the `true-stuck-{platform}-{client_slug}` shape. On the 2026-05-20 natural fire the 5 P1 true-stuck findings emitted cleanly while the 2 P2 findings (`zero-counts-pub-published-30m`, `s17-escalation-rate`) had no derivable condition_key and were routed to `friction.emit_error` with `CONDITION-KEY-UNRESOLVED` — function-contract drift (the function gained `skipped_count` + the pattern-matcher after the v3.0 lock).

**Option A (ratified, A-005 `56e992b4`):** brief author supplies an explicit per-finding `condition_key`; the function consumes it verbatim. No Supabase migration, no function patch, no re-emission. Forward-only fix via brief v3.0 → v3.1.

## CCD read-only verification finding (the v3.1 defect)

CCD's read-only verification established that the live emission path has **TWO gates**, and v3.1 satisfied only the first:

1. ✅ `friction.fn_emit_health_check_findings` **DOES consume an explicit `finding.condition_key` first** — the v3.1 input-layer design was correct.
2. ❌ `friction.emit_event` then requires an **enabled `friction.emission_rule` row for `(source='health_check', condition_key)`**, and the only currently-enabled health_check rule is **`condition_key = true_stuck`**.

Consequently **all 9 v3.1 mapping keys would have been rejected at the emission_rule gate — including the renamed `true_stuck_cluster`** (the live-enabled key is `true_stuck`, not `true_stuck_cluster`). A v3.1 fire would likely have produced `success_count=0` with `failure_count=5–7`, **regressing the P1 true-stuck emissions that have worked since 2026-05-17**.

This is the lesson candidate **L-v3.03-a**: verify downstream acceptance gates (e.g. `emission_rule`), not just the function input contract, before declaring a contract-reconciliation patch correct.
