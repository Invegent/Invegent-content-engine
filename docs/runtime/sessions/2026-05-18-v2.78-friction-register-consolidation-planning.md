## Honest limitations of this session

- **Planning was extensive, execution is zero.** 32 decisions locked; cc-0017a not authored yet. Risk is that "we have a great plan" becomes "we don't actually ship it for weeks". Mitigation: cc-0017a authoring should happen in next session when PK directs.
- **Multi-LLM review was 3 reviewers, not 4 or 5.** Higher review count would have surfaced more findings. But returns diminish after 3 — first 3 already produced strong convergence. Going to 5+ is over-investment.
- **No production validation of dedupe trigger fix yet.** Decision #23 + #25 specify the fix but the actual `fn_promote_event_to_case` rewrite happens in Wave 0b. Until then we're trusting the empirical pattern (22 events / 22 cases / max=1) is the actual current state.
- **Cross-source dedupe is deferred to v2** — this means two emitters detecting the same underlying problem from different angles (sentinel + doctor both flagging stuck-IG) will create two cases under v1. Operationally OK because they'll be in the same /operations queue, but not deduped at the case level. Worth tracking as a v2 priority.
- **Wave 0a/0b/0c is still a substantial single delivery arc.** Even split into three, cc-0017a alone has ~7 new columns on friction.case + 4 new tables. cc-0017b has the new trigger + emit_event function + concurrency tests. This is meaty work that may take 2-3 sessions per sub-wave.
- **The 1-3 LLM reviews could not validate empirical state.** Only chat-side with Supabase MCP could verify the dedupe gap; reviewers worked from the doc + reasoning. Reviewers explicitly noted this limitation (Review 3 GitHub-mode noted Supabase was unavailable on refresh).
