## Context — why the reframe

The v2.92 WARN was named "Cowork scheduling cadence WARN" because the brief was producing one output file roughly every 5 calendar days since v3.0 published 2026-05-15. The implicit hypothesis was: scheduler is firing sparsely; root-cause Cowork agent uptime / cron misconfiguration / task paused.

External observation supplied with this directive corrects that framing:

- **16/19 days fire** for `nightly-health-check-v1` over the relevant window.
- **14/19 canonical 16:00 UTC fires.**

That is not primarily a scheduling problem. The scheduler is largely working. The defect lives downstream: when the brief fires cleanly and emits findings, the resulting `status: review_required` blocks the next scheduled fire — because the queue's lifecycle had no path back to `ready` for recurring briefs that simply produced output.

The 2026-05-17 run is the test case:
- Clean execution: all 14 SQL queries verbatim, 0 schema-drift fallbacks.
- Emission to `friction.event` succeeded (`success_count=5, failure_count=0`).
- V-C3 (cc-0014 Stage C live verification) UNBLOCKED.
- 9-of-10 success thresholds hit. Q-004 logged.
- Outcome: brief moved to `review_required` and **stayed** there until v2.94.

Operational findings are an in-band signal from the brief. They belong in `friction.case` triage at `/operations`. They do not belong as a hard block on the recurring brief's next fire.

### 3. Convention patched

`docs/runtime/automation_v1_spec.md` Status flow § (line ~174):

- Status-flow diagram extended with a return arrow indicating recurring briefs can route back to `ready`.
- Valid-transitions list extended with two new transitions:
  - `review_required → ready` (recurring brief: PK reviewed any pending findings/questions and the next scheduled fire is allowed to pick up the brief)
  - `done → ready` (recurring brief next-fire reset where the prior cycle ended at `done`)
- New "Recurring-brief lifecycle" subsection codifies 4-case routing:
  1. **Clean execution + operational findings** → route findings to downstream sink (e.g. `friction.event` via v3.0 dual-write); set brief back to `status: ready`. PK approval not required to advance the brief itself; PK approval is required to act on the findings (separate sink).
  2. **Execution / schema / emission failure** → set brief to `status: review_required` (original "needs PK eyes" path).
  3. **Brief-language ambiguity / questions** → set brief to `status: questions_pending` (or co-exist with `ready` for recurring briefs where the question doesn't block the next fire). A logged Q-NNN does NOT by itself force `review_required`.
  4. **One-shot briefs** → legacy `review_required → done` terminal transition is unchanged.

Operational principle (new): a clean recurring-brief run that produces findings should never sit at `review_required` and gate the next fire. If it does, treat it as a process bug — reset to `ready`, route any open question to the question queue, and triage findings in the downstream sink.
