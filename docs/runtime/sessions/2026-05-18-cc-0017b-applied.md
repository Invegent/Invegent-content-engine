### Corrective migration: `cc_0017b_emit_event_ambiguity_fix`

**Defect discovered:** V-B10 (happy-path emit_event call) raised Postgres error code **42702**: `column reference event_id is ambiguous`. Cause: `RETURNS TABLE (event_id uuid, ...)` creates implicit OUT parameter `event_id` that shadows the column reference in `UPDATE friction.event SET case_id = v_case_id WHERE event_id = v_event_id` (Step 9 of emit_event body, line 164 of migration).

**Blast radius if left unfixed:**
- All 3 wrappers broken (reconciliation/health_check/manual)
- cron 85 daily fire (03:30 AEST) would silently log `friction.emit_error` and produce no events
- cron 86 daily fire (Cowork health check at 01:15 UTC) same failure
- FAB clicks from dashboard would error out to user

**Fix:** One-line qualification `WHERE friction.event.event_id = v_event_id` (schema-qualified column reference disambiguates from OUT param).

**D-01 fire for corrective migration:** `a6415afa-...`. Verdict=partial, type-(b) corrected_action: "evaluate comprehensive ambiguity check before execution". Chat performed line-by-line ambiguity audit of entire emit_event body — confirmed Step 9 is the ONLY ambiguity (all other DB references are aliased via CTE names or schema-qualified). Path A satisfy-corrected-action chosen by PK. Corrective migration applied successfully.
