## 1. Problem

Full mechanism documented in `docs/audit/runs/2026-05-03-baudit-v4-peers.md` § "Active-path findings → F-AAP-002". Not re-litigated here. One-line:

`m.run_system_audit` Check 7 (`orphaned_active_post_drafts`) flags `m.post_draft` rows where `pd.digest_item_id` doesn't link to a valid `m.digest_item` as data integrity warnings. Under v4, slot-driven drafts have `pd.digest_item_id = NULL` by design — this is normal, not orphaned. With F-AAP-001 now applied and v4 drafts approving, the audit will start reporting ~100+ "orphaned" drafts every Sunday at 13:00 UTC (cron jobid 43 weekly).

**Severity:** P2. Not a pipeline break. Audit produces wrong verdicts; reading the audit log will lie to the reader. Doesn't propagate wrong values into mutations.

**Active path:** jobid 43 weekly. Currently silent because audit only runs weekly and v4 drafts couldn't approve until F-AAP-001 fix landed. **Next misfire window: 2026-05-10 Sunday 13:00 UTC** unless this fix is applied first.

## 2. Fix design — tighten the gate

The current Check 7:

```sql
WITH orphaned AS (
  SELECT COUNT(*) as cnt FROM m.post_draft pd
  LEFT JOIN m.digest_item di ON di.digest_item_id = pd.digest_item_id
  WHERE di.digest_item_id IS NULL
    AND pd.approval_status IN ('needs_review', 'approved')
)
```

The fix adds one condition to the WHERE clause:

```sql
WITH orphaned AS (
  SELECT COUNT(*) as cnt FROM m.post_draft pd
  LEFT JOIN m.digest_item di ON di.digest_item_id = pd.digest_item_id
  WHERE di.digest_item_id IS NULL
    AND pd.slot_id IS NULL                         -- NEW: exclude v4 slot-driven drafts
    AND pd.approval_status IN ('needs_review', 'approved')
)
```

Also update the recommendation SQL string in the same check (so the runbook query a human runs to investigate orphans matches the gate logic):

```sql
-- Current recommendation string:
'SELECT post_draft_id, approval_status FROM m.post_draft pd
 LEFT JOIN m.digest_item di ON di.digest_item_id = pd.digest_item_id
 WHERE di.digest_item_id IS NULL AND pd.approval_status IN (''needs_review'',''approved'')'

-- Updated recommendation string:
'SELECT post_draft_id, approval_status FROM m.post_draft pd
 LEFT JOIN m.digest_item di ON di.digest_item_id = pd.digest_item_id
 WHERE di.digest_item_id IS NULL AND pd.slot_id IS NULL AND pd.approval_status IN (''needs_review'',''approved'')'
```

**Why `pd.slot_id IS NULL` is the right discriminator:** confirmed via `m.fill_pending_slots` source-read in B-AUDIT-V4-PEERS. v4 slot-driven drafts are inserted with `slot_id = v_slot.slot_id` (NOT NULL). Legacy v3 drafts are inserted by `m.create_post_drafts_v1` and other v3 paths with `slot_id` defaulting to NULL. The two populations are cleanly separable on `slot_id` nullability.

**Preserved:** SECURITY DEFINER, search_path, language, parameter list, RETURNS TABLE signature, all 11 other RETURN QUERY blocks (operational/data_integrity/compliance/structural checks), category and invariant_name strings for Check 7, status logic (`pass` vs `warn`), detail string format, recommendation string structure.
