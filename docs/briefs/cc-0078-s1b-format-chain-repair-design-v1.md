## 1. The chain — one defect in three links

| Link | Line | Current behaviour | Role |
|---|---|---|---|
| **Cause** | `:404` | `const { data: formatRows } = await supabase.rpc("exec_sql", …)` — **`error` discarded** | a DB failure becomes `undefined` |
| **Amplifier** | `:427` | `formats = allFormats.length > 0 ? allFormats : [{ ice_format_key: 'text', … }]` | palette collapses to a **hardcoded single `text` entry** |
| **Enforcer** | `:472` | `formatKey = validKeys.includes(parsed?.format_key ?? '') ? parsed.format_key : 'text'` | any out-of-palette answer coerced to **`'text'`** |

**One silent DB failure at `:404` produces a guaranteed `text` output on every platform — including
YouTube, where text is unpublishable.** The Advisor is *told* the palette is text-only, so it
"correctly" chooses `text`; nothing reports a fault. The result looks like an editorial decision.

**Why one repair, not two.** `:427`/`:472` were cc-0079 Slice 1's scope — the **symptom**. `:404` is
`cc-0078`'s — the **cause**. Landing Slice 1 alone replaces *"silently always text"* with *"silently
plausible-but-wrong"*: the output stops looking obviously degraded while still being built from a
palette the database never supplied. **That is worse, because it is harder to notice.** One writer,
one change, one test suite.

## 2. 🔒 The `:472` framing — carried verbatim, and it must survive review

> **Coercing an out-of-palette LLM answer to a valid key is CORRECT model-output validation and must
> be preserved.** The model must not invent format keys. The defect is (a) coercing to a **hardcoded
> literal** rather than a platform-valid member of the real palette, and (b) being unable to
> distinguish *"the model went out of range"* from *"the palette was never loaded."*
> **The repair preserves the validation and removes the literal.**

**This paragraph exists so a future reviewer does not "simplify" the validation away.** A reviewer
seeing `:472` in a diff about "removing hardcoded text fallbacks" could reasonably conclude the
coercion itself is the bug. It is not. Deleting it would let a hallucinated `format_key` reach
`writeVisualSpec` and the downstream render path.

## 3. Target behaviour

### 3.1 `:404` — the palette load becomes three-state

| State | Condition | Behaviour |
|---|---|---|
| `failed(error)` | query errored | **FAIL CLOSED** — the job errors and is retryable. Never a silent text-only palette. |
| `configured-empty` | query succeeded, zero buildable formats for this client+platform | proceed on a **platform-valid** default, **explicitly recorded** |
| `loaded(rows)` | normal | normal |

**Fail-closed is the right trade here** (`cc-0078` v1 §3, unchanged by the compliance park): a
retryable job error is better than silently degrading every client to text-only. **The pipeline will
fail more often and more visibly. That is the intent.**

### 3.2 `:427` — the literal goes, the fallback becomes platform-aware

The hardcoded `[{ ice_format_key: 'text', … }]` is replaced by a **platform-valid** default derived
from the same source of truth the palette comes from — never an inline literal. **`text` is a valid
default for Facebook and LinkedIn; it is not a valid default for YouTube**, which is the whole reason
this is a product defect and not a cosmetic one.

**The default must be platform-derived, not a second hardcoded table.** A per-platform literal map
inside `ai-worker` would recreate the same defect one layer up — the platform-support data already
lives in `t."5.3_content_format".platform_support`, which the `:404` query already selects and `:422`
already parses. **Where the default comes from is unpinned** (`cc-0052` L5); a plausible route is the
platform-filtered palette itself, but the implementation lane chooses and justifies.

⚠ **Open question if a platform has no buildable format at all (§6 Q2)** — that is `configured-empty`
for that platform, and silently substituting `text` there would reintroduce exactly this bug.

### 3.3 `:472` — validation preserved, literal removed, cases separated

Coercion stays. It coerces to a **platform-valid member of the actual palette**, and the two
conditions become distinguishable:

- **model went out of range** (palette loaded, answer invalid) → coerce + record `advisor_out_of_range`
- **palette never loaded** → cannot occur post-repair; `:404` fails closed first

## 4. Test requirements

- **Error injection, not empty injection.** Every touched site gets a test injecting
  **`{data: null, error: <PostgrestError>}`** — *not* `{data: []}`. **Mocking only `data` is exactly
  what let `cc-0052` ship.**
- **A distinct test per state**: `failed` · `configured-empty` · `loaded`. A two-state suite cannot
  catch a two-state regression.
- **Platform matrix.** For each live platform, assert (a) the chosen format is platform-valid, and
  (b) a `:404` failure **halts** rather than yielding `text`. **The YouTube case is the headline: it
  must be impossible for a DB failure to produce a text draft for YouTube.**
- **`:472` regression guard:** a test asserting an out-of-palette model answer is still coerced —
  so a future "simplification" that deletes the validation fails CI.
- **Live verification** (`cc-0052` L3): a real post-deploy generation on a non-text-capable platform.
  Unit tests cannot prove live PostgREST behaviour.

## 5. Blast radius, and what makes this safe to ship

**Touched:** the format-context loader and the advisor's validation step in `ai-worker`.
**Not touched:** the compliance loader (S-1a, **PARKED**) · the shadow suggester (`:320`/`:334`) ·
advisory sites (`:430`/`:436`/`:989`) · `heygen-worker` · any DB object · any schema.

**The change is contained to how the palette is loaded and validated.** It alters *which formats are
available* only in the failure case — where today the answer is unconditionally wrong.

⚠ **Honest risk: this changes production content selection.** On the happy path the palette is
identical, so drafts should be unchanged — but "should" is exactly the kind of claim `cc-0052`
punished. **Acceptance requires observed live output, not a parity argument** (§4).

⚠ **It will surface latent DB fragility as visible job failures.** If `:404` has been failing
intermittently and silently, this repair converts those into errors. **That is the point, and PK
should expect it** — a spike in job failures after deploy is evidence the repair works, not that it
broke something. Distinguishing the two requires the telemetry in §3.1 to ship *with* the fix.

## 6. Open questions for PK

**Q1 — Confirm fail-closed for `:404`.** A DB error halts the job rather than degrading to text. This
is the availability trade; it is smaller than the compliance one (a failed draft, not unsafe content),
but it is still a trade.

**Q2 — What if a platform genuinely has zero buildable formats?** That is `configured-empty` for that
platform. Options: fail closed as a config defect, or fall back to the client's other platforms'
capability. **Silently substituting `text` is not an option — it is this bug.**

**Q3 — Sequencing against step B.** Both are `ai-worker`/`heygen-worker` T3 lanes owned by S3. They
touch **different files** (`ai-worker` vs `heygen-worker`) so they do not collide, but each needs its
own gate and its own deploy window. **Recommend: S-1b first** — it is the near-term product fix with
the narrower blast radius, and step B's design is the strategically larger piece.

**Q4 — Does S-1b ship the shared wrapper?** The `cc-0078` design put the wrapper in S-1a, which is now
parked. S-1b needs *some* error-checked read at `:404`. **Recommend: S-1b introduces the wrapper**
(built once, reused when S-1a unparks) rather than a one-off `if (error)` that S-1a would later
replace — otherwise the parked slice guarantees a second edit to the same lines.
