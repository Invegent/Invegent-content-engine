## 1. The `:404 → :427 → :472` chain — one defect, verified in source

PK's causal chain is confirmed at `800a02e`, and it is **one chain, not two defects**:

| Link | Line | Code | Effect |
|---|---|---|---|
| **Cause** | `:404` | `const { data: formatRows } = await supabase.rpc("exec_sql", …)` | DB error discarded → `formatRows` undefined |
| **Amplifier** | `:427` | `const formats = allFormats.length > 0 ? allFormats : [{ ice_format_key: 'text', … }]` | palette collapses to a **hardcoded single `text` entry** |
| **Enforcer** | `:472` | `const formatKey = validKeys.includes(parsed?.format_key ?? '') ? parsed.format_key : 'text'` | any answer outside the (now text-only) palette is **coerced to `'text'`** |

**Consequence:** one silent DB failure at `:404` produces a **guaranteed `text` output on every
platform — including YouTube, where text is unpublishable.** The advisor is even *told* the palette is
text-only, so it "correctly" chooses text; nothing anywhere reports a fault.

**Why they must land together.** Slice 1 scoped `:427`/`:472` — the **symptom**. cc-0078 owns `:404` —
the **cause**. Fixing `:427`/`:472` alone (platform-aware literals) would replace *"silently always
text"* with *"silently plausible-but-wrong"* — a **worse** failure, because the output stops looking
obviously degraded while still being produced from a palette the database never supplied. **One
writer, one change, one test suite.**

⚠ **`:472` is not purely a defect.** Coercing an out-of-palette answer to a valid key is correct
*model-output* validation — the LLM must not invent format keys. The defect is that it coerces to a
**hardcoded literal** rather than to a **platform-valid member of the real palette**, and that it
cannot tell "the model went out of range" from "the palette was never loaded". The repair preserves
the validation and removes the literal.

## 2. The three-state contract (PK Ruling 1)

> **A query error, a legitimately empty configuration, and a loaded result must be three distinct,
> non-interchangeable values.**

Today all three collapse to `''` (compliance) or `[]` (palette), and generation proceeds. **That
collapse is the defect** — not the emptiness itself.

| State | Meaning | Compliance behaviour | Palette behaviour |
|---|---|---|---|
| `failed(error)` | the database did not answer | **FAIL CLOSED — halt generation.** A failed job is the correct outcome. | **FAIL CLOSED** — retryable job error |
| `configured-empty` | genuinely no rules/formats configured for this client | **PROCEED, explicitly recorded** as a deliberate zero | proceed on a **platform-valid** default, recorded |
| `loaded(rows)` | normal path | normal | normal |

**Recording `configured-empty` is mandatory, not optional.** A zero that is *known to be a real
configuration state* is materially different from a zero of unknown provenance — and today's data
cannot distinguish them (§4). The repair must make that distinction durable from the moment it ships.

**Availability trade accepted by PK:** expect failed jobs where today there are silently-unconstrained
drafts. **Stated plainly: this lane will make the pipeline fail more often. That is the point.**

## 3. Repair design — properties, not a pinned mechanism

Mechanism deliberately **not pinned** (`cc-0052` L5: *"a pinned mechanism in a brief is a risk
concentrator"*). The implementation lane selects one against these properties.

### 3.1 Required properties

- **P1 — the error cannot be discarded.** The helper's return type must make ignoring `error`
  *unrepresentable*, not merely discouraged. A shape that can still be destructured to a bare array
  fails this property.
- **P2 — three states, distinguishable at every call site** (§2).
- **P3 — each call site declares fail-closed or fail-open explicitly.** Today the decision is made
  accidentally and always in favour of "empty".
- **P4 — a failure emits distinguishable telemetry**, not a bare `console.error` no pipeline reads.
- **P5 — no new string-interpolated SQL.** 10 of 11 sites interpolate; this lane must not add an
  eleventh. The existing injection shape stays OPEN and accepted-for-now (`cc-0052` §9) and is **not**
  this lane's subject.
- **P6 — currently-benign sites keep their behaviour** unless deliberately changed, so the diff's
  blast radius stays inspectable.
- **P7 — the existing `try/catch` wrappers are treated as part of the defect** (census §2.3): a
  discarded error never throws, so the `catch` never runs, and both paths converge on the same silent
  fallback. Removing the discard without revisiting the catch leaves the same hole.

### 3.2 Shared wrapper, not 11 hand-edits

Eleven independent `if (error)` insertions are eleven chances to diverge and prevent nothing at site
twelve. One helper that **cannot return a bare array** makes the defect structurally unrepresentable.
**One contract across both workers** — a second competing compliance wrapper is explicitly forbidden
(PK Ruling 2).

### 3.3 Staging — split the governance decision, not the wrapper

| Slice | Sites | Content | Gate |
|---|---|---|---|
| **S-1a** ⭐ | `:488`, `:501`, `:511` | **Compliance three-state, fail-closed.** The first separately-gated T3 slice. | own T3 gate |
| **S-1b** | `:404`, `:427`, `:472` | **The format chain + absorbed Slice 1**, as ONE change | own T3 gate |
| **S-2** | `:92`, `:455` | `heygen-worker`. Own window; **real live submit** proof (`cc-0052` L3). | own T3 gate |
| **S-3** | `:430`, `:436`, `:989`, `:320`, `:334` | Advisory + shadow: reason-code and telemetry correctness | own gate |

**The wrapper itself is built once, in S-1a**, and reused unchanged. S-1a and S-1b are separately
gated because their *risk decisions* differ (halting generation vs changing format selection) — not
because their mechanism differs.

**Sequencing constraints:** `heygen-worker` keeps **one writer** — if any other lane opens it, S-2
goes first or waits. **S3 and S7 must not edit `ai-worker` concurrently**; S7 holds Slice 2
(data-only mix renormalization) and is consulted as reviewer on format behaviour.

### 3.4 Test requirements

- Every repaired site: a test injecting **`{data: null, error: <PostgrestError>}`** — *not* `{data: []}`
  — asserting the declared behaviour. **Mocking only `data` is exactly what let `cc-0052` ship.**
- **A distinct test per state**: `failed` · `configured-empty` · `loaded`. A two-state test suite
  cannot detect a two-state regression.
- **S-1b needs a platform matrix test** — for each platform, assert the chosen format is
  platform-valid, and assert that a `:404` failure **halts** rather than yielding `text`.
- **S-2 requires a real live submit.** Unit tests cannot exercise live PostgREST resolution.

## 4. Forensic — compliance exposure (PK Ruling 4)

**Bounded, read-only. Nothing mutated, nothing republished.**

**A durable trail exists** — better than expected. `ai-worker` persists
`draft_format -> 'ai' -> 'compliance_rules_injected'` (written at `:1107` from the count computed at
`:683`). So the compliance-rule count is recorded **per draft**.

### 4.1 What the data shows

| Metric | Value |
|---|---|
| Drafts carrying the field | **2108** (2026-03-20 → 2026-07-23) |
| Drafts with `rules > 0` | **2068** — ndis-yarns 957 · property-pulse 726 · invegent 195 · care-for-welfare 190 |
| **Drafts with `rules = 0`** | **40 — all `property-pulse`** |
| Zero-rule window | **2026-03-20 13:10Z → 2026-04-11 13:05Z**, then never again |
| Zero-rule drafts after 2026-04-11, any client | **0** |
| Distinct rule counts observed | 0, 3, 20, 22 |
| Drafts **without** the field at all | **489** (2026-02-11 → 2026-03-19) |

### 4.2 The shape is inconsistent with sporadic DB failure

Four independent signals point the same way:

1. **Zero interleaving.** **No client-day has both zero-rule and rules>0 drafts.** A sporadic DB error
   would scatter randomly through the timeline; this does not.
2. **Clean cutover.** Last zero-rule draft 2026-04-11 13:05Z; first PP draft with `rules>0`
   2026-04-12 13:05Z — the very next daily run. A step change, not a fault distribution.
3. **Never recurred** across four clients and three further months.
4. **Not a missing-scope state.** PP's `client_content_scope` rows were created **2026-03-01** —
   *before* the zero window opened. So `:488` was resolving scope; the zero came from **`:511`
   matching no active rule** for PP's verticals (`au-mortgage-lending`, `au-property-investment`,
   `au-residential-property`), which today match exactly **3** active rules — consistent with PP's
   observed count of 3 after the cutover.

**Assessment: those 40 drafts are far more consistent with `configured-empty` — real rule-coverage
absence for PP's verticals — than with database failure.** Something changed rule coverage or the
filter on ~2026-04-11/12; no new rules were created then (`t."5.7_compliance_rule"` max `created_at`
is 2026-03-20), so it was an `is_active`/`vertical_slug` update or a code change. **I did not chase
that further — it is outside this lane and does not change the design.**

### 4.3 ⚠ Evidentiary limits — stated, not glossed

**I cannot prove those 40 were not DB errors, and I will not claim it.** The field records `0` for
*both* states — **that is precisely the defect this lane exists to fix.** The shape argument above is
strong circumstantial evidence, not proof. Specifically:

- **489 drafts (2026-02-11 → 2026-03-19) carry no field at all** — pre-instrumentation. **Their
  compliance exposure is unknowable from this data.** That is a genuine blind spot, not a clean bill.
- **A zero-rule draft is not necessarily unsafe content** — it means the *rules were not injected*,
  not that a rule was violated. Assessing the 40 drafts' actual content is a separate question this
  lane did not attempt and does not authorise.
- **`:501`/`:989`/`:436`-class degradations leave no trace at all.** A draft generated with
  *narrowed* rules (profession filter silently dropped) records a plausible non-zero count and is
  **indistinguishable from a correct one.** The forensic can only see the total-zero case.
- **Per PK's instruction I am not declaring "no affected drafts."** The system's failure to log a
  condition is not evidence the condition did not occur — the trap the census itself named.

**Honest summary: 40 zero-rule drafts exist, all PP, all in one bounded window, most consistent with
a real configuration gap; and an unmeasurable population before 2026-03-20.** Nothing was mutated or
republished.

### 4.4 What the forensic proves about the design

The three-state contract is **retro-justified by this data**: had `configured-empty` been recorded
distinctly from `failed` in March, this question would have been answerable in one query instead of
an inference chain. **The recording requirement in §2 is not bookkeeping — it is what makes the next
incident diagnosable.**
