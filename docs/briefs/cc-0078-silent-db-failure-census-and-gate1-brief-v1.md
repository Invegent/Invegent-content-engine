## 1. Verdict up front — the lane is bigger than avatars

Three findings change what `cc-0078` is:

1. **The census is 11 sites, not 8.** `ai-worker` has **9**, not 8 — `:404` uses double-quoted
   `"exec_sql"` and is invisible to a `'exec_sql'` grep. `heygen-worker` has **2** (`:92`, `:455`).
   **Zero of the eleven check `error`.**
2. 🔴 **The most dangerous sites are not the avatar lookup — they are the compliance loader.**
   `fetchComplianceBlock` (`:488`, `:501`, `:511`) returns **`''` — an empty string — on any database
   error**, and the caller then generates content with **no compliance rules at all**. No HARD_BLOCK,
   no SOFT_WARN, no exception, no log line the pipeline reads. For NDIS/health-adjacent content that
   is a **content-safety** consequence, not a reliability one.
3. **The pattern's true cost is misattribution, not just failure.** Every one of these sites converts
   *"the database did not answer"* into *"the answer is nothing"* — and downstream code then reports
   a **specific, confident, wrong reason**: `no_persona_available`, `no_active_brand_roles`,
   "No active avatar", or silently, an empty compliance block. **The system does not know it is
   broken; it reports a plausible business condition instead.**

**Recommended repair (§6): a shared wrapper, not 11 hand-edits, in ONE lane covering both workers.**

### 2.3 Why the enclosing `try/catch` does not save these

`fetchFormatContext` and `fetchComplianceBlock` are both wrapped in `try { … } catch { return <empty
or fallback> }`. That looks like protection and is the opposite:

- A discarded `error` **never throws**, so the `catch` never runs. The failure passes *through* the
  guard as a legitimate-looking empty result.
- Both the error path and the catch path **converge on the same silent fallback** — text-only
  formats, empty compliance. So even a thrown error produces the identical degraded output with only
  a `console.error` that no pipeline check reads.

**The catch blocks are load-bearing in appearance only.** Any repair must treat them as part of the
defect, not as mitigation.

## 3. The contract: DB failure ≠ genuine no-result

The single requirement this lane exists to impose:

> **A query error and an empty result must never be representable by the same value.**

Every site above collapses `{data: null, error: <PostgrestError>}` and `{data: [], error: null}` into
the same downstream state. Any repair must produce a **three-way** outcome — `ok(rows)` ·
`empty` · `failed(error)` — and each call site must **decide explicitly** which of `empty` and
`failed` it can tolerate. Today that decision is made accidentally, and always in favour of "empty".

**Fail-closed vs fail-open is per-site, and PK's call, not the repair's:**

- **Compliance (`:488`/`:501`/`:511`) must fail CLOSED** — a DB error must **abort the generation**,
  never emit unconstrained content. Generating without compliance rules is worse than not generating.
- **Format palette (`:404`) should fail CLOSED** — better a retryable job error than silently
  degrading every client to text-only.
- **Avatar (`:92`) should fail LOUD** — distinguish "no avatar configured" (a real business state)
  from "the lookup failed" (retryable). The 2026-07-23 outage message said the former and meant the
  latter.
- **Advisory sites (`:430`, `:436`, `:989`, `:455`) may fail OPEN** — but must **log a distinguishable
  telemetry event**, not a bare `console.error`.
- **Shadow sites (`:320`, `:334`) must record a distinct reason code** (e.g. `persona_query_failed` vs
  `no_persona_available`) — otherwise they keep poisoning the diagnostic record they exist to produce.

## 4. Observability — why none of this is visible today

**Submit-phase failures write NO `m.post_render_log` row, by design** — no `provider_job_id` exists
yet, and the telemetry is scoped to terminal render outcomes (`cc-0052` §4). So an avatar-lookup
failure surfaces as a **stuck or failed draft**, never as an alert. The `ai-worker` sites are worse:
they do not even fail — they produce *plausible content* with reduced constraints, so there is no
stuck draft to notice.

**What would make them visible** (named, not designed — the repair lane specifies):

1. **A submit-phase failure record.** Anything durable written when a submit aborts before a provider
   job exists. Without it, the whole class stays invisible to log-based auditing — and an auditor
   checking `post_render_log` will find nothing and wrongly conclude no outage occurred.
2. **A distinguishable DB-failure counter/event per call site** — so "the query failed" is countable
   and separable from "there were no rows".
3. **A compliance-block-empty alarm.** An empty compliance block on a client that *has* configured
   scope is either a DB failure or a config error; **neither should be silent**. This is the single
   highest-value alarm in the list.
4. **A format-palette-collapsed alarm** — palette size 1 (`text` only) for a client with
   `client_format_config` rows means `:404` failed.
5. **Reason-code separation in the shadow suggester** (§3) so the persona lane's own diagnosis stops
   being corrupted by this pattern.

## 5. Regression tests + the standing prohibition

**Regression-test requirement.** Every repaired site needs a test that injects
`{data: null, error: <PostgrestError>}` — **not** `{data: []}` — and asserts the site's declared
behaviour (abort / distinct reason code / logged event). Unit tests that mock only `data` are exactly
what let `cc-0052` ship: *"unit tests cannot exercise live PostgREST resolution."* At least one live
end-to-end exercise is required for the avatar path (`cc-0052` L3: verify against a **real live
submit**, never a parity claim).

**Standing prohibition (proposed for ratification):** **no new string-interpolated SQL may be
introduced** in `heygen-worker` or `ai-worker` by this or any successor lane. 10 of the 11 sites
interpolate caller-supplied values directly. This lane does **not** fix that — it must not *add* to
it. The existing injection shape stays OPEN and accepted-for-now (service-role only, not
anon-reachable, `cc-0052` §9) and is **not** this lane's subject.

## 6. Recommended repair — shared wrapper, one lane, staged

**Shared wrapper, not 11 hand-edits.** A per-site `if (error)` is 11 chances to be inconsistent and
leaves nothing preventing site 12. A single helper — one that **cannot return a bare array**, forcing
each caller to handle the failure case — makes the defect structurally unrepresentable. The mechanism
is **deliberately not pinned** (`cc-0052` L5): the repair lane chooses it against these properties.

**Required properties:** error is never discardable · `empty` and `failed` are distinct values · each
call site declares fail-closed or fail-open · a failure emits distinguishable telemetry · no new
string interpolation · behaviour at every currently-benign site is unchanged unless deliberately
altered.

**One lane or per-worker? — ONE lane for the wrapper, staged application.** The contract must be
identical across both workers or the pattern regrows. But the *application* should be staged by risk,
each stage its own gate:

| Stage | Sites | Why this order |
|---|---|---|
| **S-1** | `:488`, `:501`, `:511`, `:404` | The CRITICAL four. Compliance and capability loss are the real exposure, and they are **not** what the lane was opened for. |
| **S-2** | `:92`, `:455` | The avatar path. `heygen-worker` is the file with the outage history — it gets its own window and a **real live submit** proof. |
| **S-3** | `:430`, `:436`, `:989`, `:320`, `:334` | Advisory + shadow. Lowest risk; mostly reason-code and telemetry correctness. |

**Tier: T3** when implemented — production Edge Function code on the content-generation and render
paths, requiring deploy. Full chain + external review pinned to the diff hash + PK deploy gate +
rollback proven + `--no-verify-jwt` preserved. **This census lane is read-only and authorises none of
it.**

**Sequencing constraint:** S-2 touches `heygen-worker`, which must keep **one writer**. If any other
lane opens that file, S-2 goes first or waits — never concurrently.
