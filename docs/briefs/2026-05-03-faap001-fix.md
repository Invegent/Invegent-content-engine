## 1. Problem

Full mechanism documented in `docs/runtime/sessions/2026-05-03-faap001-rootcause.md`. Not re-litigated here. One-line:

`m.auto_approver_fetch_drafts(p_limit)` INNER-JOINs `m.digest_item` + `m.digest_run`. Slot-driven v4 drafts (`slot_fill_synthesis_v1` job type, ai-worker v2.10.0+, shipped late-April per slot-driven Phase A 26 April) have `pd.digest_item_id = NULL` by design — the slot-fill mechanism never inserts a `m.digest_item` row. INNER JOIN drops every v4 draft. **0 of 122** current `needs_review` drafts pass the fetcher. Auto-approver runs every 10min via cron jobid 58 and no-ops.

Date inflection: oldest NULL-`digest_item_id` `needs_review` draft `created_at` = 2026-04-27 01:40 UTC; 131 of 135 (97%) post-25-April drafts have NULL `digest_item_id`. Last system-wide approval: 2026-05-02 19:00 UTC (legacy digest-flow backlog exhausted).

**Severity:** P1. Blocks auto-approver across the entire v4 architecture.

## 2. Fix design — Path 1 (loosen the SQL fetcher)

Path 1 chosen over Path 2 (populate `digest_item_id` at slot-fill) per v2.27 recommendation. Path 1 removes a stale dependency the v4 architecture is intentionally trying to drop; smaller diff; no upstream surface area.

Rewrite `m.auto_approver_fetch_drafts(p_limit integer)` to:

- **JOINs.** Change `INNER JOIN m.digest_item di ON di.digest_item_id = pd.digest_item_id` → `LEFT JOIN`. Same for `INNER JOIN m.digest_run dr ON dr.digest_run_id = di.digest_run_id` → `LEFT JOIN`.
- **client_id source.** Read from `pd.client_id` directly. v3 drafts have both `pd.client_id` and `dr.client_id`; v4 drafts only have `pd.client_id`. Pre-flight check #3 below verifies these match for v3 drafts (data integrity).
- **final_score.** Continue reading from `di.final_score` (will be NULL for v4 drafts via LEFT JOIN). Existing `ORDER BY ... NULLS LAST` handles correctly. D135 already made `final_score` behaviourally a no-op, so v4 drafts intermixing with v3 drafts at NULL final_score is acceptable — ordering is no longer load-bearing.
- **LATERAL on `c.client_publish_profile`.** Already keyed by `pd.client_id` and `pd.platform` per source-read in v2.27. No change required.
- **All other filters / CTEs / cooldown logic / status checks.** Preserve unchanged. Pre-flight #4 below requires CC to scan the full captured source for any other reference to `dr.*` or `di.*` and evaluate each under LEFT JOIN NULL semantics.
