## Why R6 exists

R5 landed `m.match_demand_to_canonicals()` — a read-only function that says "for this client this week, here's which canonical should fill each demand slot." It produces 20 rows for NY, 20 for PP, 8 for CFW, 10 for Invegent.

**R5's output is not wired into the hot path.** The current `m.seed_and_enqueue_ai_jobs_v1()` function still picks candidates the old way: "latest digest_items bundled+success+selected, ranked by created_at." It ignores R5 entirely.

R6 rewrites `seed_and_enqueue` to consume R5's output. When R6 ships, R5's matching decisions become the actual drafts being generated. The router track finally touches the publishing pipeline.

R6 also bundles three other fixes that naturally land in the same surface:
- **Router audit Finding 1** — hardcoded client UUIDs in `m.enqueue_publish_from_ai_job_v1` trigger
- **Router audit Finding 4** — hardcoded demand formula in seeder
- **Router audit Finding 6** — `NOT IN ('youtube')` exclusion

- **R7 (ai-worker platform awareness)** — separate piece, happens after R6 proves out

## Task A — Rewrite `m.seed_and_enqueue_ai_jobs_v1` (CC)

### Current behaviour (to be replaced)

File at: `sql/functions/m.seed_and_enqueue_ai_jobs_v1.sql` (if extracted) or live via `pg_get_functiondef('m.seed_and_enqueue_ai_jobs_v1(text,integer)'::regprocedure)`.

What it does today (simplified):
1. For each active client with `publish_enabled=true` and `mode='auto'` on the platform, compute a target_depth (current hardcoded formula: `CEIL(schedule_slots * 1.5)` or `CEIL(max_per_day * 7 * 1.5)`)
2. Count what's already in pipeline (`post_draft` with status in draft/needs_review/approved/scheduled in the last 7 days)
3. Allowance = max(0, target_depth - in_pipeline)
4. Pick digest_items (latest bundled, success, selected) up to allowance per client, capped by `p_limit` overall
5. Insert seed → insert draft stub → insert ai_job

### New behaviour (contract)

**Function signature unchanged:**
```sql
m.seed_and_enqueue_ai_jobs_v1(
  p_platform TEXT DEFAULT 'facebook',
  p_limit INT DEFAULT 10
) RETURNS jsonb
```

Keeping the signature stable means the crons don't need to change (Task D is just command text polish, not contract change).

**New algorithm:**
1. For each active client with `publish_enabled=true`, `mode='auto'` on `p_platform`, AND `r6_enabled=true`:
   - Call `m.match_demand_to_canonicals(client_id, CURRENT_DATE, 50.0, 1)` → gets up to 20 rows (varies by client demand grid)
   - Filter matches to `WHERE platform = p_platform` (R5 returns all platforms; cron is still per-platform)
   - Also filter out matches whose `digest_item_id` already has a `post_seed` for this `(client, platform)` — R5 already does this check but be defensive (idempotency)
2. Aggregate candidates across all eligible clients, cap at `p_limit` globally (lowest-scored candidates drop first)
3. For each winning candidate, insert the row chain:
   - `m.post_seed` — with `seed_payload` enriched (see schema below)
   - `m.post_draft` — with `recommended_format` populated from R5 output (was NULL before)
   - `m.ai_job` — unchanged logic

**Enriched `seed_payload` schema (NEW):**
```json
{
  "platform": "facebook",
  "digest_item": { ...existing digest_item jsonb... },
  "match_metadata": {
    "ice_format_key": "carousel",
    "content_class": "multi_point",
    "fitness_score": 98.0,
    "quality_score": 75.0,
    "recency_score": 70.0,
    "final_match_score": 82.7,
    "fitness_version": "v1",
    "weights_version": "global_default_40_30_30",
    "match_reason": "fitness=98.0 × 0.40 + quality=75.0 × 0.30 + recency=70.0 × 0.30 = 82.7 (fitness_source=default, weights=global_default_40_30_30, within_run_dedup=greedy_v1.3)",
    "matched_at": "2026-04-24T11:45:00Z"
  }
}
```

This metadata is how R7 will eventually learn which match decisions produced good engagement — it's the audit trail.

**New `post_draft` column writes (NEW):**
- `recommended_format TEXT` ← from R5's `ice_format_key`. Was NULL in old seeder.

**Everything else — `client_id` on draft, seed_type, ai_job rows — unchanged.**

### Behaviour when R5 returns empty

If `m.match_demand_to_canonicals()` returns zero rows for a client (demand grid empty, or pool genuinely can't fill anything), R6 just skips that client silently. No seed, no draft, no job. Don't log errors — R5 returning empty is a legitimate state surfaced by the diagnostic, not an error.

### Return payload (extended)

Same top-level shape as before — add a `matches_considered` field:
```json
{
  "ok": true,
  "version": "r6-matching-driven-v1",
  "platform": "facebook",
  "seeded": 5,
  "drafts": 5,
  "jobs_queued": 5,
  "matches_considered": 12,     // new: total candidates from R5 before p_limit cap
  "clients_processed": 2,        // new: active clients with r6_enabled=true
  "clients_skipped_gated": 2     // new: active clients blocked by r6_enabled=false
}
```

The new fields help Task C verify expected behaviour and help future debugging.

### Idempotency requirements

Function is called every 10 min via cron. Must be safe to call back-to-back without producing duplicate seeds/drafts/jobs.

- **Seed uniqueness:** rely on `post_seed_uniq_run_item_platform` unique constraint `(digest_run_id, digest_item_id, platform)` with `ON CONFLICT DO NOTHING`
- **Draft uniqueness:** `NOT EXISTS` check on `(digest_item_id, platform)` — same as current
- **Job uniqueness:** `NOT EXISTS` check on `(post_draft_id, job_type)` — same as current

## Task F — Rewrite trigger (here)

### Why this stays here

This trigger fires on EVERY ai_job success, inserting into publish queue. Any bug produces real client content issues within the next cron cycle (10 min). Live-fire territory. I want eyes on every line.

### New trigger body (abbreviated)

Replace the 9-branch hardcoded UUID cascade with column reads:

```sql
CREATE OR REPLACE FUNCTION m.enqueue_publish_from_ai_job_v1()
  RETURNS trigger
  LANGUAGE plpgsql
  SECURITY DEFINER
  SET search_path TO 'pg_catalog', 'public', 'm'
AS $function$
DECLARE
  v_queued_count INT;
  v_max_queued INT;
  v_min_gap INTERVAL;
  v_last_sched TIMESTAMPTZ;
  v_next_sched TIMESTAMPTZ;
BEGIN
  IF tg_op = 'UPDATE'
     AND new.status = 'succeeded'
     AND (old.status IS DISTINCT FROM new.status)
     AND COALESCE(new.platform, '') IN ('facebook', 'linkedin', 'instagram')
     AND new.post_draft_id IS NOT NULL
     AND new.client_id IS NOT NULL
  THEN
    -- Read cap + gap from client_publish_profile (R6: Finding 1)
    SELECT 
      COALESCE(cpp.max_queued_per_platform, 10),
      COALESCE(
        cpp.min_post_gap_minutes_override * INTERVAL '1 minute',
        cpp.min_gap_minutes * INTERVAL '1 minute',
        INTERVAL '120 minutes'  -- ultimate fallback
      )
    INTO v_max_queued, v_min_gap
    FROM c.client_publish_profile cpp
    WHERE cpp.client_id = new.client_id
      AND cpp.platform = new.platform;

    -- If no row found, use defaults
    v_max_queued := COALESCE(v_max_queued, 10);
    v_min_gap := COALESCE(v_min_gap, INTERVAL '120 minutes');

    SELECT COUNT(*) INTO v_queued_count
    FROM m.post_publish_queue
    WHERE status = 'queued'
      AND client_id = new.client_id
      AND platform = new.platform;

    IF v_queued_count < v_max_queued THEN
      SELECT MAX(scheduled_for) INTO v_last_sched
      FROM m.post_publish_queue
      WHERE status = 'queued'
        AND client_id = new.client_id
        AND platform = new.platform
        AND scheduled_for IS NOT NULL;

      v_next_sched := GREATEST(NOW(), COALESCE(v_last_sched, NOW()) + v_min_gap);

      INSERT INTO m.post_publish_queue (
        ai_job_id, post_draft_id, client_id, platform, scheduled_for, status, attempt_count
      )
      VALUES (
        new.ai_job_id, new.post_draft_id, new.client_id, new.platform, v_next_sched, 'queued', 0
      )
      ON CONFLICT (post_draft_id, platform) DO NOTHING;
    END IF;
  END IF;

  RETURN new;
END;
$function$;

COMMENT ON FUNCTION m.enqueue_publish_from_ai_job_v1() IS
  'R6 version (2026-04-24): replaced hardcoded client UUID cascade with reads from c.client_publish_profile.max_queued_per_platform + min_post_gap_minutes_override. Closes Finding 1. Behaviour preserved exactly for NY + PP via backfill values.';
```

### Verification after deploy

Run this query to confirm the new trigger produces identical cap/gap decisions as the old:
```sql
SELECT 
  cpp.client_id,
  c.client_slug,
  cpp.platform,
  cpp.max_queued_per_platform,
  COALESCE(cpp.min_post_gap_minutes_override, cpp.min_gap_minutes) AS effective_gap_minutes
FROM c.client_publish_profile cpp
JOIN c.client c ON c.client_id = cpp.client_id
WHERE cpp.platform IN ('facebook', 'linkedin', 'instagram')
  AND c.client_slug IN ('ndis-yarns', 'property-pulse')
ORDER BY c.client_slug, cpp.platform;
```

Expected output matches the hardcoded trigger cascade exactly.

**Q3 — Should the pool-adequacy diagnostic auto-gate?**
Radical option: `r6_enabled` column is replaced by a function that checks `m.vw_match_pool_adequacy.adequacy_grade IN ('A','B')`. Removes the manual flip step. Risk: a feed goes down, grade drops to C, R6 silently stops producing for that client.
**Default answer:** Manual flip. Auto-gating hides too much. Explicit is better.

**Q4 — Fallback behaviour when R5 returns empty slots?**
Current spec: skip empty slots silently. Alternative: log a row to a new `m.match_empty_slot_log` table per skip, for diagnostic visibility.
**Default answer:** Log to a new table. The adequacy diagnostic tells you in aggregate; this would tell you per-run which slots missed. ~15 extra min on Task A.

**Q5 — What happens if ALL clients are gated off?**
Function returns `{ok: true, seeded: 0, clients_processed: 0, clients_skipped_gated: 4}`. Cron fires every 10 min producing this no-op. Is that noise worth logging to m.cron_health_alert?
**Default answer:** No — this is the designed behaviour during soft-launch of R6. Not an alert condition.
