## 1. What is live today (verified in source at `052a9ba`)

`heygen-worker` `runSubmitPhase`, the single identity-selection site:

| Line | Behaviour |
|---|---|
| `:427` | `talkingPhotoId = fmt.talking_photo_id ?? fmt.avatar_id ?? null` — a **preset** short-circuit |
| `:432` | `if (!talkingPhotoId)` — the lookup runs **only** when no preset exists |
| `:433` | `lookupAvatar(supabase, clientId, stakeholderRole, renderStyle)` |
| `:434` | `if (!avatar) throw new Error("No active <style> avatar for client <id>")` |
| `:437` | `avatarSelectedBy = stakeholderRole ? 'role_filter' : 'fallback_limit1'` |
| `:92` | inside `lookupAvatar`: `const { data: rows } = …` — **`error` discarded** (`cc-0078` D5, live) |

The query is `is_active = true` ∧ exact `render_style` ∧ optional `role_code`, then **`LIMIT 1` with
no `ORDER BY`**. Correct today **only** because each of NDIS Yarns and Property Pulse has exactly one
active avatar. **`is_default_host` is referenced nowhere.**

**Three facts step B must not disturb:**

1. **The preset short-circuit at `:427` bypasses resolution entirely.** A draft carrying
   `talking_photo_id` never reaches the resolver. Step B changes *resolution*, not that precedence —
   unless PK rules otherwise (§7 Q3).
2. **`avatar_selected_by` is observability-only** and copied verbatim into `avatar_identity` and then
   `post_render_log.render_spec`. Adding a value is additive; **changing an existing value's meaning
   is not.**
3. **`stakeholder_role` is null on all 86 drafts** — the role predicate has never fired in production.
   Step B must not make the role path load-bearing on a path with zero production evidence.

## 2. The resolution order — designation first, deterministic throughout

**Replace the unordered `LIMIT 1` with a total order whose first key is the governed designation.**

```
candidates := active ∧ render_style = <style> ∧ (role_code = <role> when role given)
order by:
  1. is_default_host DESC   -- the governed designation (step A's output) — NEW, load-bearing
  2. is_primary     DESC    -- role-canonical marker (0 rows set; inert but ordered)
  3. created_at     ASC     -- stable tiebreak
  4. brand_avatar_id ASC    -- total order — never ties
limit 1
```

This is **exactly the order the dormant shadow resolver already uses**
(`agp-d01-3-shadow-resolver-telemetry.md` §B.2). Step B does not invent an ordering; it **promotes the
already-designed one into the live path.** Reusing it means the shadow resolver and the live resolver
finally agree by construction rather than by coincidence.

**Behaviour on today's data: byte-identical.** Each candidate set is size 1, so any total order
returns the same row. **That is the acceptance baseline (§5), not a reason to skip proving it.**

**What changes materially:** the *first* moment a second avatar becomes active, selection is decided
by a **declared fact** instead of storage order. Step B is the difference between "correct by luck"
and "correct by decision".

## 3. The no-designation paths — governed, not accidental

`cc-0047` step 4 requires Invegent and CFW to resolve to an **explicit governed `no-eligible-avatar`
state, never a silent fall-through.** Four distinct outcomes, each with its own reason code:

| # | Condition | Outcome | `avatar_selected_by` |
|---|---|---|---|
| **1** | candidates exist, exactly one designated | select it | `default_host` |
| **2** | candidates exist, **none designated** | select by the §2 tiebreak, **flag as ungoverned** | `undesignated_tiebreak` |
| **3** | **zero candidates** (Invegent, CFW) | **fail closed, explicit** | `no_eligible_avatar` |
| **4** | the query **errored** | **fail closed, distinct** | `resolution_failed` |

**Case 2 is the one that needs stating.** 26 of 28 rows are undesignated. If a client's only active
avatar is undesignated, refusing to render would be an outage manufactured by governance. So it
**proceeds** — but it is **recorded as ungoverned**, never silently equated with a governed pick.
This mirrors the `cc-0078` four-state contract: *ungoverned-but-proceeding must be visible.*

**Cases 3 and 4 must be distinguishable — this is the whole point.** Today both produce the same
`"No active avatar"` throw, which is precisely how the 2026-07-23 outage was misread as a data
condition. **Never again collapse "no avatar configured" (a real business state, permanent until an
avatar is onboarded) with "the lookup failed" (transient, retryable).**

⚠ **Case 3 is the live state for Invegent and CFW right now.** Neither has an avatar, so neither can
run `video_short_avatar` today. Step B does not fix that — it makes the refusal **legible** instead of
an exception that reads like a fault. Onboarding those clients is a separate lane (carry C-1).

## 4. `cc-0052` lessons — binding constraints, not references

The lane that last touched this exact function caused a production outage. Its lessons are design
requirements here:

- **L1 — check `error` on every query.** The current `:92` **discards it** (`cc-0078` D5, still live).
  Step B touches this function, so **D5 is repaired as part of step B** — a null result and an errored
  result become cases 3 and 4 above, never the same value. **This is the one place the parked
  `cc-0078` work is not parked: it is unavoidable here, because the defect is inside the function
  step B rewrites.** Scope is exactly `:92`, nothing else in the census.
- **L2 — no "strict parity" assumption across execution paths.** `cc-0052` asserted parity between an
  `exec_sql` string join and a query-builder `!inner` embed on the **non-`public` `c` schema**, and the
  embed does not resolve via supabase-js at runtime. **Any mechanism must be proven against the live
  path, and the `c`-schema embed specifically is presumed unsafe until demonstrated otherwise.**
- **L3 — verify against a real live submit**, never a parity claim plus unit tests. Unit tests cannot
  exercise live PostgREST resolution. §5 requires it.
- **L5 — mechanism deliberately NOT pinned.** `cc-0052`'s brief pinned the query-builder and
  de-scoped the RPC, removing the fallback that would have avoided the outage. **Candidates stay open:
  a parameterised SECDEF RPC (mirroring the proven shadow-resolver posture) · a corrected
  query-builder form · an ordered `exec_sql` with `error` checked.** The implementation lane chooses
  and justifies; **the RPC option must remain on the table.**

**Required properties:** error is never discardable · the four outcomes are distinct values · the
order is total (never ties) · `is_active` semantics unchanged · no new string interpolation · the
preset short-circuit's precedence unchanged unless PK rules otherwise.

## 5. Acceptance — parity first, then governance

**Stage 1 — prove no behaviour change (the baseline).**
On current data every candidate set is size 1, so step B **must** select the identical avatar for both
brands. Evidence: a **real live `video_short_avatar` submit** (`cc-0052` L3) selecting the same
`talking_photo_id`/`voice_id` as before. Per `cc-0063` Q4 precedent, **prefer a naturally-scheduled
submit; do not manufacture one** unless PK elects otherwise.

**Stage 2 — prove the designation is actually consumed.** Parity alone cannot distinguish "reads the
designation" from "still reads storage order" — both give the same answer on size-1 sets. **This is
the `declared-control-not-consulted` trap, and step B is the lane that must not fall into it.**
Acceptable proofs, PK's choice (§7 Q2):
- `avatar_selected_by = 'default_host'` appearing on a live draft — a positive signal that the
  designation branch executed; **or**
- a read-only replay comparing live vs designation-ordered selection over a constructed multi-row
  candidate set, **without activating anything**.

**What is NOT acceptable:** inspection of the diff, unit tests alone, or "the code says it reads it."

**Stage 3 — negative-path evidence:** cases 3 and 4 produce **distinct** reason codes, demonstrated by
fault injection in test, and case 3 confirmed against Invegent or CFW's genuinely-zero state.

## 6. A → B → C — where this lands, and what it does not unlock

**Step B closes the precondition that step C waits on.** PK's ruling: *no second avatar may be
activated until the live resolver consumes governed host designation, or an equivalent
database-enforced ambiguity guard exists.*

⚠ **Step B satisfies that precondition by a governance route, not a database one — and PK should
approve it knowing that.** After step B, selection *among candidates* is governed and deterministic.
But **C-2 remains open**: the dashboard's `assign_brand_avatar` still performs an unconditional
`UPDATE … SET is_active = true` with **no per-(client, render_style) guard**, so an operator can still
create a second active avatar in one click. Step B makes that **safe to survive** — the resolver would
pick the designated host deterministically — but it does **not prevent** the second avatar appearing,
and it does not make the new avatar designated.

**What step B assumes about C-2:** that an *undesignated* second avatar appearing beside a designated
one resolves to the **designated** one (case 1 beats case 2 by the §2 order). **That assumption is
what makes step B a genuine mitigation.** It holds only while the designation is present — if an
operator activated a second avatar **and** designated it, the partial unique index would reject the
second designation, which is the DB-enforced half of the guard.

**C-2 is S1's scoped finding and stays there.** Step B neither fixes nor absorbs it (standing rule:
findings are recorded and carried, not absorbed).

**Step C remains CLOSED** until PK opens it as its own gate. **Step B does not authorise activating a
second avatar, and completing it does not make step C automatic.**
