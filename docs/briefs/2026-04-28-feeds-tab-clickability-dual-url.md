## 1. Why this brief exists

Brief A (`2026-04-28-feeds-tab-discovery-context.md`) shipped earlier today. It surfaced feed URLs and discovery context on the Feeds tab. Two follow-up gaps emerged in PK's V8 review:

1. **The unassigned bucket renders the URL as a clickable link. The assigned bucket renders it as plain text.** Inconsistent. Same data, different click behaviour for no good reason.

2. **For URL-type discovery seeds, only the rss.app endpoint URL is currently surfaced.** The original site URL the rss.app feed was generated from (e.g. `https://www.ndis.gov.au/news`) is the more useful link for an operator who wants to verify the source content. Right now to get to it the operator has to click the rss.app URL → land on the rss.app feed page → click through to the source. Two clicks where one would do.

3. **Source_name for URL-type seeds IS the URL.** Migration 007 backfilled `source_name = seed_value`. For keyword seeds this is a sensible label (e.g. "sensory gym", "paediatric occupational therapy"). For URL seeds the source_name becomes the URL itself (e.g. "https://www.ndis.gov.au/news"). The row reads ugly in the dashboard. A friendly hostname-derived name would read cleaner.

## 3. Target state

### 3.1 URL clickability — both buckets

The assigned-bucket URL line becomes a clickable `<a target="_blank" rel="noopener noreferrer">` exactly mirroring Brief A's unassigned-bucket pattern. Same truncation (60 chars), same colour (`text-blue-500 dark:text-blue-400`), same hover (`hover:text-blue-700 hover:underline`).

### 3.2 Dual-URL display for URL-type discovery seeds

Where a URL-type discovery seed exists, render two URL lines:

1. **Original site URL** (the seed_value) — primary, labeled "Source"
2. **rss.app feed URL** — secondary, labeled "Feed", smaller font

Both clickable, both `target="_blank"`. For keyword seeds, only the rss.app URL renders (no source URL exists). For operator-added rss_app feeds (no discovery seed), only the rss.app URL renders. For rss_native feeds, only the native URL renders (no separate "feed" — they're the same thing).

Visual sketch:

```
NDIS Government News                    [Auto-discovered]
Source: https://www.ndis.gov.au/news
Feed:   https://rss.app/feeds/JUeglyzwYBvtRcjs.xml
```

### 3.3 Source_name polish for URL-type seeds

Migration 007's backfill set `source_name = seed_value` uniformly. For URL seeds this leaves source_name as a raw URL.

Replace with hostname-derived friendly name. Reuse existing `seed_value` so we don't lose information — just rename `source_name`.

**Naming function logic:**

Given a URL like `https://www.ndis.gov.au/news`:
1. Extract hostname → `www.ndis.gov.au`
2. Strip leading `www.` → `ndis.gov.au`
3. Take leftmost label → `ndis`
4. Suffix with path-tail descriptor if path is non-trivial:
   - `/news` → "news"
   - `/blog` → "blog"
   - `/feed` → (omit — redundant for a feed)
   - `/` → (omit)
   - other → "updates" (catchall)
5. Combine → `"NDIS news"` (Title Case the hostname leftmost label, lowercase the path-tail)

Examples for verification:

| Input URL | Output source_name |
|---|---|
| `https://www.ndis.gov.au/news` | `NDIS news` |
| `https://www.otaus.com.au/news` | `Otaus news` |
| `https://example.com/blog/posts` | `Example blog` |
| `https://acme.org/feed` | `Acme` |
| `https://multi-word-domain.io/` | `Multi-word-domain` |

Coverage:
- All future URL seeds get this name at write time (new branch in `add_client_discovery_seeds` RPC)
- Existing URL-seed-derived rows in `f.feed_source` get a one-time backfill UPDATE
- Keyword seeds untouched (their source_name = keyword is already the right label)
- Operator-added rss_app/rss_native feeds untouched (source_name was operator-supplied)

This is achievable with a `f_url_to_friendly_name(text)` PL/pgSQL function — no need to call out to anything external.

## 4. Implementation plan

### 4.1 Migration `009` — friendly-name function + backfill

In `Invegent-content-engine`. CC writes the file, applies via Supabase MCP `apply_migration`, commits the .sql file to the repo.

```sql
-- Function: derive a friendly source_name from a URL
CREATE OR REPLACE FUNCTION public.f_url_to_friendly_name(p_url text)
RETURNS text
LANGUAGE plpgsql
IMMUTABLE
AS $$
DECLARE
  v_host text;
  v_path text;
  v_label text;
  v_tail text;
  v_path_descriptor text;
BEGIN
  IF p_url IS NULL OR p_url !~* '^https?://' THEN
    RETURN p_url;
  END IF;

  -- Strip scheme, split on first /
  v_host := lower(regexp_replace(p_url, '^https?://', ''));
  v_path := '';
  IF position('/' in v_host) > 0 THEN
    v_path := substring(v_host from position('/' in v_host));
    v_host := substring(v_host from 1 for position('/' in v_host) - 1);
  END IF;

  -- Strip www.
  v_host := regexp_replace(v_host, '^www\.', '');

  -- Leftmost label, capitalise
  v_label := initcap(split_part(v_host, '.', 1));

  -- Path tail descriptor
  v_tail := lower(trim(both '/' from v_path));
  IF v_tail IN ('', 'feed', 'rss', 'feeds') THEN
    v_path_descriptor := '';
  ELSIF v_tail = 'news' THEN
    v_path_descriptor := ' news';
  ELSIF v_tail LIKE 'news/%' OR v_tail LIKE 'news?%' THEN
    v_path_descriptor := ' news';
  ELSIF v_tail = 'blog' THEN
    v_path_descriptor := ' blog';
  ELSIF v_tail LIKE 'blog/%' OR v_tail LIKE 'blog?%' THEN
    v_path_descriptor := ' blog';
  ELSE
    v_path_descriptor := '';
  END IF;

  RETURN v_label || v_path_descriptor;
END;
$$;

GRANT EXECUTE ON FUNCTION public.f_url_to_friendly_name(text) TO authenticated, service_role, anon;

-- Backfill: rename source_name for URL-derived discovery rows
-- Only touch rows where:
--   - originated from a URL-type seed (via discovery_pipeline)
--   - source_name still LOOKS like a URL (starts with http)
UPDATE f.feed_source fs
SET source_name = public.f_url_to_friendly_name(fs.source_name)
WHERE fs.added_by = 'discovery_pipeline'
  AND fs.source_name ~* '^https?://';

-- Update add_client_discovery_seeds RPC: for URL seeds, store the friendly name as label
-- (label is what the EF picks as source_name via fallback chain)
-- This means going forward, new URL seeds land with friendly name automatically.
CREATE OR REPLACE FUNCTION public.add_client_discovery_seeds(
  p_client_id    uuid,
  p_keywords     text[] DEFAULT '{}'::text[],
  p_example_urls text[] DEFAULT '{}'::text[]
)
RETURNS jsonb
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public, f, c, t
AS $$
DECLARE
  v_vertical_slug text;
  v_keyword text;
  v_url text;
  v_inserted_count int := 0;
  v_skipped_count int := 0;
BEGIN
  -- (validation block unchanged from migration 007)
  IF NOT EXISTS (
    SELECT 1 FROM c.client
    WHERE client_id = p_client_id AND status = 'active'
  ) THEN
    RAISE EXCEPTION 'Client % not found or not active', p_client_id;
  END IF;

  IF (array_length(p_keywords, 1) IS NULL OR array_length(p_keywords, 1) = 0)
     AND (array_length(p_example_urls, 1) IS NULL OR array_length(p_example_urls, 1) = 0)
  THEN
    RAISE EXCEPTION 'Must provide at least one keyword or example URL';
  END IF;

  SELECT cv.vertical_slug
    INTO v_vertical_slug
  FROM c.client_content_scope ccs
  JOIN t.content_vertical cv ON cv.vertical_id = ccs.vertical_id
  WHERE ccs.client_id = p_client_id
  ORDER BY ccs.is_primary DESC NULLS LAST,
           COALESCE(ccs.weight, 1.0) DESC,
           cv.vertical_slug ASC
  LIMIT 1;

  IF v_vertical_slug IS NULL THEN
    v_vertical_slug := 'general';
  END IF;

  -- Keywords: label = trim(keyword), unchanged from migration 007
  FOREACH v_keyword IN ARRAY p_keywords
  LOOP
    IF length(trim(v_keyword)) = 0 THEN
      v_skipped_count := v_skipped_count + 1;
      CONTINUE;
    END IF;

    IF EXISTS (
      SELECT 1 FROM f.feed_discovery_seed
      WHERE client_id = p_client_id
        AND seed_type = 'keyword'
        AND lower(seed_value) = lower(trim(v_keyword))
    ) THEN
      v_skipped_count := v_skipped_count + 1;
      CONTINUE;
    END IF;

    INSERT INTO f.feed_discovery_seed
      (seed_type, seed_value, vertical_slug, status, client_id, label)
    VALUES
      ('keyword', trim(v_keyword), v_vertical_slug, 'pending', p_client_id,
       trim(v_keyword));

    v_inserted_count := v_inserted_count + 1;
  END LOOP;

  -- URLs: label = friendly name (NEW — was trim(v_url) in migration 007)
  FOREACH v_url IN ARRAY p_example_urls
  LOOP
    IF length(trim(v_url)) = 0 THEN
      v_skipped_count := v_skipped_count + 1;
      CONTINUE;
    END IF;

    IF trim(v_url) !~* '^https?://' THEN
      v_skipped_count := v_skipped_count + 1;
      CONTINUE;
    END IF;

    IF EXISTS (
      SELECT 1 FROM f.feed_discovery_seed
      WHERE client_id = p_client_id
        AND seed_type = 'url'
        AND lower(seed_value) = lower(trim(v_url))
    ) THEN
      v_skipped_count := v_skipped_count + 1;
      CONTINUE;
    END IF;

    INSERT INTO f.feed_discovery_seed
      (seed_type, seed_value, vertical_slug, status, client_id, label)
    VALUES
      ('url', trim(v_url), v_vertical_slug, 'pending', p_client_id,
       public.f_url_to_friendly_name(trim(v_url)));

    v_inserted_count := v_inserted_count + 1;
  END LOOP;

  RETURN jsonb_build_object(
    'inserted', v_inserted_count,
    'skipped',  v_skipped_count,
    'vertical_slug', v_vertical_slug,
    'next_discovery_run_utc', '20:00 daily'
  );
END;
$$;
```

### 4.3 URL display helper

In `feeds-client.tsx`. Add a small helper that, given a FeedRow, returns the URLs to render:

```typescript
type FeedUrls = {
  source: string | null;       // Original site URL (only for URL-type discovery seeds)
  feed: string | null;         // The rss.app/native feed URL
};

function getFeedUrls(feed: FeedRow): FeedUrls {
  // For URL-type discovery seeds: source = seed_value, feed = source_url
  if (feed.discovery_seed_type === 'url' && feed.discovery_seed_value) {
    return {
      source: feed.discovery_seed_value,
      feed: feed.source_url ?? null,
    };
  }
  // Everything else: only the feed URL is meaningful
  return {
    source: null,
    feed: feed.source_url ?? null,
  };
}
```

### 4.4 Render upgrade — assigned bucket

In `feeds-client.tsx`, find the assigned-bucket render block (the `feed.source_url && <p>...` line). Replace with the dual-URL pattern:

```tsx
{(() => {
  const urls = getFeedUrls(feed);
  return (
    <div className="mt-0.5 space-y-0.5">
      {urls.source && (
        <a
          href={urls.source}
          target="_blank"
          rel="noopener noreferrer"
          className="text-xs text-blue-500 dark:text-blue-400 hover:text-blue-700 dark:hover:text-blue-300 hover:underline font-mono truncate block"
        >
          <span className="text-slate-400 mr-1">Source:</span>
          {urls.source.length > 60 ? urls.source.slice(0, 60) + '\u2026' : urls.source}
        </a>
      )}
      {urls.feed && (
        <a
          href={urls.feed}
          target="_blank"
          rel="noopener noreferrer"
          className="text-xs text-blue-500 dark:text-blue-400 hover:text-blue-700 dark:hover:text-blue-300 hover:underline font-mono truncate block"
        >
          {urls.source && <span className="text-slate-400 mr-1">Feed:</span>}
          {urls.feed.length > 60 ? urls.feed.slice(0, 60) + '\u2026' : urls.feed}
        </a>
      )}
    </div>
  );
})()}
```

If only one URL exists, the "Source:" / "Feed:" prefix labels do not render — keeps the visual clean for the common case. The labels only appear when both are present (URL-type discovery seeds).
