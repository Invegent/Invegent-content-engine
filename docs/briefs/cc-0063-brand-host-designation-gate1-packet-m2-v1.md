**Properties of that file, verified by review (§13):**

- **One column, two rows.** The only `SET` clause in the file is `SET is_default_host = true`
  (line 68); the `UPDATE` is predicated on an explicit `brand_avatar_id IN (…)` of the two named
  UUIDs. Statically verified — no upsert, no `ON CONFLICT`, no unqualified `UPDATE`.
- **One atomic `DO` block, no explicit `BEGIN`/`COMMIT`.** Deliberate: a `DO` block is a single
  statement, so it is atomic under **either** apply path — `apply_migration` (which supplies its own
  transaction) or `execute_sql` at a PK gate. An explicit `BEGIN` can conflict with a caller-supplied
  transaction; this form depends on neither. Any `RAISE` aborts the whole statement **including the
  `UPDATE`**. This is a deliberate improvement on the earlier three-block form, which relied on the
  wrapper's `BEGIN`/`COMMIT` actually being honoured.
- **Three guards:** entry fingerprint (0 pre-existing markers, exactly 2 active) → `GET DIAGNOSTICS`
  row-count assertion (exactly 2) → post-write invariants re-`SELECT`ed **after** the `UPDATE` inside
  the same statement (designated total = 2 · zero duplicate `(client, render_style)` groups ·
  `is_active` still 2 · zero designated rows that are not active-realistic).
- **No `EXCEPTION` handler anywhere**, so no `RAISE` can be swallowed.
- **`IS TRUE` throughout** — `c.brand_avatar.is_active` is **nullable**; `IS TRUE` excludes NULL
  unambiguously and agrees with live `lookupAvatar`'s `is_active = true`, which also excludes NULL.
- **Deliberately not idempotent** — a re-run aborts at Guard 1. Endorsed by PK 2026-07-24.

⚠ **Clean-replay hazard (recorded, accepted):** this is a data migration with hard-coded UUIDs and
entry-state assertions. Against an empty or shadow database (`supabase db push`, a CI shadow-diff,
any replay-based workflow) it **raises at Guard 1** — "expected exactly 2 active avatars, found 0".
Low impact for ICE, which applies against production only, but it permanently poisons replay
workflows. Recorded here rather than patched into the file, so the reviewed hash stays stable.

**Deliberate properties**

- **Not idempotent, on purpose.** The `is_default_host = false` predicate means a second run affects
  0 rows and **raises**. A silent no-op re-run would be indistinguishable from success; a loud abort
  is correct.
- **No `ON CONFLICT`, no upsert, no unqualified `UPDATE`.** Explicit `brand_avatar_id IN (…)`.
- **`is_active IS TRUE`, not `= true`** — the column is **nullable** (§2), and `IS TRUE` excludes NULL
  unambiguously. The live `lookupAvatar` uses `is_active = true`, which also excludes NULL; the
  predicates agree, and the assertion counts use `IS TRUE` consistently so a NULL can never be
  miscounted as active.
- **Guard 3 proves `is_active` untouched from inside the transaction**, so a violation rolls back
  rather than being caught after the fact.
